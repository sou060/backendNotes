Follow-up Traps

These are the questions where a confident first answer invites a chain of follow-ups. Interviewers use them to see how deep your understanding actually goes — rehearse the whole chain, not just the opener.

"How does HashMap work internally?"

  1. Opener: You explain buckets, hashCode(), and equals().
  2. Follow-up 1: "What happens when two keys hash to the same bucket?" — Collision handling: a linked list of entries, which Java 8+ converts to a red-black tree once a bucket holds more than 8 entries (treeify threshold), turning O(n) worst case into O(log n).
  3. Follow-up 2: "What happens during resize?" — When size exceeds capacity × load factor (default 0.75), the table doubles and every entry is rehashed into new bucket positions — an O(n) operation, which is why pre-sizing a HashMap when you know the count helps performance.
  4. Follow-up 3: "Is HashMap thread-safe? What breaks if two threads resize concurrently?" — No. Concurrent structural modification (like simultaneous resizes) can corrupt the bucket's linked list into a cycle, causing an infinite loop on get() in older JDKs. Use ConcurrentHashMap instead.

"What does the volatile keyword do?"

  1. Opener: Guarantees visibility of writes across threads by preventing caching in CPU registers/thread-local memory.
  2. Follow-up 1: "Does volatile make a compound operation like count++ thread-safe?" — No. Visibility isn't atomicity; count++ is read-modify-write, and two threads can still interleave those three steps. You'd need AtomicInteger or synchronization.
  3. Follow-up 2: "So when is volatile actually enough on its own?" — For simple flags (e.g., a boolean `running` a worker thread polls to stop), where only one thread ever writes and others only read.
  4. Follow-up 3: "How does volatile relate to the Java Memory Model's happens-before relationship?" — A write to a volatile field happens-before every subsequent read of that field, which also makes all writes that happened before the volatile write visible to the reading thread — this is how double-checked locking singleton patterns rely on volatile.

"Why is String immutable in Java?"

  1. Opener: Enables safe sharing via the string pool, thread-safety by default, and safe use as a HashMap key.
  2. Follow-up 1: "If Strings are immutable, why does `str = str + "x"` appear to modify it?" — It doesn't; it creates a new String object and rebinds the reference `str` to point to it. The original object is unchanged and becomes eligible for garbage collection if nothing else references it.
  3. Follow-up 2: "Why is that expensive in a loop, and what's the fix?" — Each concatenation allocates a new object, making an n-iteration loop O(n²) in the worst case. StringBuilder mutates an internal char buffer instead, making it O(n).
  4. Follow-up 3: "How would you make your own class immutable like String?" — Make fields private and final, provide no setters, don't expose mutable internal state directly (return defensive copies of arrays/collections), and don't allow subclassing (mark the class final) so a subclass can't break the invariant.

"What's the difference between synchronized and ReentrantLock?"

  1. Opener: synchronized is a built-in keyword with automatic lock release; ReentrantLock is an explicit class from java.util.concurrent.locks offering more control.
  2. Follow-up 1: "What can ReentrantLock do that synchronized can't?" — tryLock() with a timeout (avoiding indefinite blocking), interruptible lock acquisition, fairness policies (FIFO ordering of waiting threads), and multiple Condition objects per lock instead of just one implicit monitor.
  3. Follow-up 2: "What's the catch with ReentrantLock?" — You must manually unlock() in a finally block; forgetting to do so causes a permanent deadlock, a mistake synchronized's automatic release makes impossible.
  4. Follow-up 3: "When would you actually reach for ReentrantLock in production code?" — When you need tryLock's non-blocking attempt (e.g., avoiding a deadlock by backing off), or when multiple distinct wait conditions on the same resource make separate Condition objects genuinely useful.

"What's the difference between ArrayList and LinkedList?"

  1. Opener: ArrayList is backed by a resizable array (O(1) random access); LinkedList is a doubly-linked list (O(1) insert/delete at known positions).
  2. Follow-up 1: "In practice, which one should you default to and why?" — ArrayList, almost always. LinkedList's O(1) insertion advantage only applies if you already hold a reference to the node; getting to an arbitrary index is still O(n), and its extra per-node object overhead (two references + object header) usually costs more in real workloads than ArrayList's occasional resize.
  3. Follow-up 2: "What actually happens when an ArrayList resizes?" — A new backing array (typically 1.5x the old capacity) is allocated and the old contents are copied over with System.arraycopy — an O(n) operation, but amortized O(1) per add() across many insertions.

"What is the diamond problem, and how does Java avoid it?"

  1. Opener: Java disallows multiple class inheritance so a class can never inherit the same method from two conflicting parents.
  2. Follow-up 1: "But interfaces can have default methods since Java 8 — doesn't the diamond problem come back?" — Yes, partially. If a class implements two interfaces with the same default method signature, the compiler forces you to explicitly override it and choose (or combine) the implementation — it refuses to guess.
  3. Follow-up 2: "How do you call one specific parent interface's default method inside your override?" — `InterfaceName.super.methodName()` — this is the one case where `super` is qualified with an interface name rather than a class.
